# FeedMe 订单控制器：实施方案

## 1. 文档目的与范围

- **目的**：在实现代码前，固定需求理解、技术选型、部署与测试策略，便于评审与迭代。
- **范围**：本文档对应仓库 [README.md](../README.md) 中的作业说明，并扩展为 **Golang 后端 + Angular 前端 + 公网部署**。正文以设计与验收标准为主；**具体实现以仓库代码为准**，随开发迭代可同步修订本文档。
- **分支**：设计与文档在仓库**默认分支 `main`** 上维护；功能开发可用特性分支，合并前向 `main` 提 PR（GitHub Flow）。
- **参考部署**：线上目标域名为 `**feedme.anycolor.ai`**，部署形态参考本地目录 `references/anycolor`（与公开站点 anycolor.ai 同源技术栈：Nginx 容器 + Traefik，详见 [线上部署说明.md](线上部署说明.md)）。

## 2. 项目目标（与作业的关系）


| 维度      | 作业原文要求         | 本项目约定                                                                         |
| ------- | -------------- | ----------------------------------------------------------------------------- |
| 实现形态    | 前端 **或** 后端二选一 | **全栈**：Angular 演示全部交互；Golang 同时提供 **HTTP API** 与 **CLI**，满足 `scripts` 与 CI 契约 |
| 后端语言    | Go 或 Node.js   | **Go**                                                                        |
| 前端      | 任意框架，需可编译部署    | **Angular**                                                                   |
| 数据      | 无需持久化          | 内存态即可                                                                         |
| 交互式 CLI | 面试下一轮必演示       | 与「`run.sh` 写 `result.txt`」拆分为不同子命令，避免互相干扰（见 [架构与接口设计.md](架构与接口设计.md)）         |


## 3. README 需求逐条对照（勿遗漏）

以下与 [README.md](../README.md) 原文一一对应，验收时按表勾选。

### 3.1 情境与用户故事


| 编号  | 原文要点                                             | 设计响应                                           |
| --- | ------------------------------------------------ | ---------------------------------------------- |
| S1  | 普通顾客：下单后进入 PENDING，机器人处理后进入 COMPLETE             | UI 分区展示；后端维护订单状态机                              |
| S2  | VIP 优先于所有普通单；已有 VIP 时新 VIP 排在队尾（FIFO within VIP） | PENDING 队列语义：**先 VIP 子队列（先到先服务），再 Normal 子队列** |
| S3  | 经理增减机器人；+Bot 立即从 PENDING 取单；-Bot 时处理中订单视为未完成     | +Bot 触发调度；-Bot **取消**当前处理并将订单按规则回到 PENDING     |
| S4  | 机器人单线程一单；每单 **10 秒**                             | 每 Bot 同时最多绑定一单；处理时长可配置，默认 10s（测试可缩短）           |


### 3.2 功能需求（Requirements 1–7）


| 编号  | 原文                                                      | 设计要点                                                                                                                                                                                            |
| --- | ------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R1  | New Normal Order → PENDING                              | API + 按钮；订单进入 Normal 子队列尾部（仍在「全体 VIP 之后」）                                                                                                                                                       |
| R2  | New VIP Order → PENDING，位于所有已有 Normal 之前、所有已有 VIP 之后    | 新 VIP 追加到 VIP 子队列尾部                                                                                                                                                                             |
| R3  | 订单号唯一且递增                                                | 全局单调递增 ID（`int64` 计数器即可）                                                                                                                                                                        |
| R4  | +Bot：创建后立即从 PENDING 按规则取单；约 10s 后 COMPLETE；有剩余则继续       | 调度器为每个空闲 Bot 分配下一单；完成后再次调度。README 中「picking up 后 10 秒」与「开始处理」对齐：**自进入 PROCESSING 起计时**（见 [实施细节补充.md](实施细节补充.md) §1）                                                                             |
| R5  | PENDING 空时 Bot IDLE，新单到达再取                              | 无单可派时 Bot 状态为空闲                                                                                                                                                                                 |
| R6  | -Bot：销毁 **最新创建**的 Bot；若处理中则停止处理，订单回到 PENDING **原优先级位置** | Bot 按创建顺序管理，**栈顶/最后一个**为「最新」；取消处理上下文；订单回到 VIP 或 Normal 子队列后，**在同类型内按订单 ID 升序插入**以恢复相对顺序。**前提**：本原型采用 **全局单调递增** 订单号，使 ID 顺序与入队先后一致；若改为 UUID 等非单调 ID，须改为「记录离队前下标」等机制（见 [架构与接口设计.md](架构与接口设计.md)） |
| R7  | 无持久化                                                    | 进程内数据结构；重启丢失可接受                                                                                                                                                                                 |


### 3.3 作业交付与脚本（Backend 部分）

**目录名说明**：作业英文 README 写作 `script` 目录；本仓库实际为 `**scripts/`**（复数）。以下均以 `scripts/` 为准。


| 条目                   | 说明                                                                                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `scripts/test.sh`    | 执行 `go test`（或项目约定测试命令）                                                                                                                                       |
| `scripts/build.sh`   | **最低义务**：编译 **CLI**（供 `run.sh` 与 CI 使用）。`cmd/server` 与 Docker 镜像可在本脚本中顺带构建，或由 `deploy` 下 Dockerfile 单独构建——团队内统一即可，但合并 PR 前须保证 **CI 中的 `build.sh` 能成功产出 CLI**。 |
| `scripts/run.sh`     | 运行 CLI（如 `feedme run-demo`），将输出写入 `scripts/result.txt`                                                                                                        |
| `scripts/result.txt` | 非空，且含 `HH:MM:SS` 时间戳（与 GitHub Actions 中 grep 规则一致）；内容应有意义（作业 Submission）                                                                                      |
| GitHub Flow          | Fork 后提 PR；合并前通过检查（详见 [持续集成说明.md](持续集成说明.md)）。工作流文件名为 `**backend-verify-result.yaml`**（作业中曾写 `go-verify-result`，以仓库为准）                                        |


**CI 覆盖范围**：当前 GitHub Actions **仅**执行 `test.sh` / `build.sh` / `run.sh` 并校验 `result.txt`，**不**执行 `ng build`。Angular 是否可编译、可部署由 [本地测试说明.md](本地测试说明.md) 或后续增补的流水线步骤保证（见 [持续集成说明.md](持续集成说明.md)）。

`**run-demo` 与耗时**：作业要求每单处理 **10 秒**；CLI 演示若按真实 wall-clock 跑多单，CI 可能变慢。允许 `run-demo` 使用 **缩短的处理时长** 或 **紧凑事件脚本**，只要 `result.txt` 仍含 `HH:MM:SS` 且能体现接单与完成时序即可。

### 3.4 前端（Angular）

- 可编译、可部署到公网；界面需能 **操作并展示** R1–R7 全部行为（PENDING / 处理中 / COMPLETE / Bot 状态 / 四个按钮）。
- 与后端通信：优先 **短轮询**（如 1s 拉取快照），降低首版复杂度；后续可改为 SSE/WebSocket（在 [架构与接口设计.md](架构与接口设计.md) 中预留）。

### 3.5 其他 README 条款

- 可使用 AI 辅助开发；**必须自行测试**（见 [本地测试说明.md](本地测试说明.md)）。
- 「30 分钟 vibe coding」为作业提示；全栈实现可分期：先核心领域与 CLI，再 API 与 UI，最后部署。

## 4. 非功能与设计原则

- **简洁**：避免不必要中间件；领域逻辑与 HTTP/CLI 解耦，便于单测复用。
- **可复用**：排队与调度集中在单一包内（如 `internal/feedme`），CLI 与 `server` 仅作适配层。
- **语言**：**对外英文**——根 README、Angular 按钮与分区标签（如 PENDING / COMPLETE）、CLI 交互提示与 `result.txt` 等面向阅卷/用户的可见文本，与 [README.md](../README.md) 英文作业一致。**对内中文**——本 `docs/` 目录全部文档与 **源码注释**（Go/TS/脚本说明性注释）。
- **安全与运维**：首版可无鉴权；公网部署时由反向代理与内网隔离 API（见部署文档）。

## 5. 里程碑建议（实现阶段，供确认）

1. **M1**：Go 模块、`internal` 领域模型与单元测试；`-Bot` 取消与回队语义单测覆盖。
2. **M2**：CLI 固定场景输出 `scripts/result.txt`（含 `HH:MM:SS`）；更新 `build.sh` / `run.sh` / `test.sh`。
3. **M3**：HTTP API + CORS；Angular 页面联调。
4. **M4**：Docker / Compose / Nginx 反代；Traefik 标签指向 `feedme.anycolor.ai`。
5. **M5**：交互式 CLI（面试用），与 demo 子命令分离。
