# 持续集成说明

## 1. 当前策略

- **GitLab 当前不可用**：本阶段 **不配置、不执行** GitLab CI/CD，也不在本文档中假设流水线已通过。
- **验证方式**：以 [本地测试说明.md](本地测试说明.md) 为准，在开发者本机完成脚本与功能验证。
- **未来恢复 CI 时**：以下章节与仓库内 GitHub Actions 对齐，作为检查清单。

## 2. GitHub Actions（仓库已存在）

工作流文件：[.github/workflows/backend-verify-result.yaml](../.github/workflows/backend-verify-result.yaml)

**覆盖范围（重要）**：该工作流 **只验证 Go 后端与 CLI 契约**（`test.sh` / `build.sh` / `run.sh` 与 `scripts/result.txt`）。**不**构建 Angular、**不**部署 Docker。全栈项目中，前端与容器化发布需在 **本地** 或 **额外 CI Job** 中验证（见下文「可选增强」）。

**触发条件**：向 `main` 的 Pull Request（`opened`、`synchronize`、`reopened`、`edited`）。

**关键步骤顺序**

1. `chmod +x scripts/test.sh scripts/build.sh scripts/run.sh`
2. `./scripts/test.sh`
3. `./scripts/build.sh`
4. `./scripts/run.sh`
5. 断言 `scripts/result.txt` 存在、非空，且含至少一处 `HH:MM:SS` 模式（正则 `[0-9]{2}:[0-9]{2}:[0-9]{2}`）。

**运行环境**

- `ubuntu-latest`
- Go（如 1.23.9）、Node（如 22.19.0）——与工作流中版本保持一致。

实现代码后，合并 PR 前应 **本地先跑通** 上述四步，避免 Actions 失败。

## 3. 作业 README 中的 workflow 名称

作业英文 README 曾提及 `go-verify-result`；本仓库实际文件名为 **`backend-verify-result.yaml`**。以仓库文件为准。

## 4. 可选增强（未来）

在不影响作业最低要求的前提下，可为同一 PR 增加例如：

- `npm ci && npm run build`（在 `frontend/` 下执行生产构建），防止 Angular 编译回归；
- `docker compose config` 或镜像构建，验证 `Dockerfile` / Compose 语法。

与现有 `backend-verify-result` 并列时需控制总时长与缓存策略。

## 5. GitLab CI（未来可选）

本地参考 **`references/anycolor/.gitlab-ci.yml`** 的模式通常为：SSH 到服务器、`git pull`、`docker compose build && up`。

- **不在此阶段实现**；恢复 GitLab 时建议：
  - 使用 **Deploy Key** 或 **CI 变量** 中的 `SSH_PRIVATE_KEY`；
  - 不在仓库中提交私钥与明文令牌。
- FeedMe 若沿用该模式，需在脚本中同时更新 **前端构建产物** 与 **API 镜像**。

## 6. 与文档、实现的关系

**hanxipu** 分支以 `docs/` 设计与验收标准为主；代码落地后应：

1. 更新 `scripts/*.sh` 调用真实二进制。
2. 本地按 [本地测试说明.md](本地测试说明.md) 验证（含 Angular 构建与 R1–R7）。
3. 再开启 PR 跑 GitHub Actions（及日后 GitLab）。
