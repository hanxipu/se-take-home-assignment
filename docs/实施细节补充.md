## 1. README 细读：易误解与已对齐点

| README 原文 | 注意点 | 本仓库约定 |
|-------------|--------|------------|
| R4：「after 10 seconds **picking up** the order」 | 可能被读成「接单后再等 10 秒」 | **从进入 PROCESSING（机器人开始处理）起计时 10 秒**，到时进入 COMPLETE；与「pickup from PENDING」同一时刻开始计时。文档与实现中写清注释。 |
| Backend：「must be a **CLI** application」 | 全栈时易被忽略 | **作业机器判分仍以 CLI + `scripts/*` 为准**；HTTP Server 为原型演示增值，**不能**替代 CLI 与 `result.txt`。 |
| `result.txt` | 未写相对路径 | 与 CI 一致，输出到 **`scripts/result.txt`**（workflow 显式校验该路径）。 |
| Submission：`go-verify-result` | 名称不一致 | 以仓库 **`.github/workflows/backend-verify-result.yaml`** 为准（已在 [持续集成说明.md](持续集成说明.md) 说明）。 |
| Frontend：「publicly accessible」 | 全栈必满足 | 通过 **`feedme.anycolor.ai`**（或作业允许的任意公网 URL）访问；本地开发不算最终交付。 |
| Tips：「30 min」「don't over engineer」 | 与全栈张力 | 分期交付（实施方案里程碑）；核心领域与 CLI 优先，UI 与部署可后补，但 **面试前** 须有可演示全链路。 |

## 2. 与 README「二选一」的合规说明（全栈）

- 作业允许 **仅前端** 或 **仅后端（CLI）**；本项目选择 **超出范围的组合** 时，建议面试/PR 说明中写清：
  - **后端路径**：Go CLI + `test.sh` / `build.sh` / `run.sh` + `scripts/result.txt` **完全满足** Backend 条款；
  - **前端路径**：Angular 构建部署，**覆盖** Frontend 条款；
  - 二者共用 `internal/feedme` 领域逻辑，避免重复实现两套规则。

## 3. HTTP API 实施约定（在草案上细化）

| 项目 | 约定 |
|------|------|
| Content-Type | `POST` 请求使用 `application/json` |
| 成功 | `POST` 返回 `201 Created`，Body 可返回创建的资源摘要（如 `{"id":1001,"kind":"vip"}`）或仅 `{"ok":true}`；团队统一即可 |
| 错误 | `400`：JSON 非法或 `kind` / `action` 非法；`405`：方法错误。`-Bot` 时若无 Bot，可返回 **`409 Conflict`** 或 **`400`**，**须在实现与前端统一**并在本文档或 OpenAPI 注释中写死 |
| 幂等 | 不要求；每次 `POST` 均创建新订单或尝试增减 Bot |
| `GET /api/state` | `200`，`Content-Type: application/json`；**禁止缓存**或 `Cache-Control: no-store`（避免轮询读到 CDN 旧页） |
| CORS | 开发：`ng serve` 与 API 不同源时，对 `localhost` 放行；生产：**同源**（Nginx 反代）可关闭 CORS |
| 可读错误体 | 若返回 JSON 内含 `error` / `message` 等给前端展示的字段，使用 **英文** |

**快照一致性**：`pending` 列表顺序建议为 **先全部 VIP（FIFO），再全部 Normal（FIFO）**，与架构文档队列语义一致，便于 UI 直接渲染而不必再排序。

| 变量 | 用途 | 建议默认 |
|------|------|----------|
| `FEEDME_LISTEN` 或 `PORT` | HTTP 监听地址 | `:8080` |
| `FEEDME_PROCESS_MS` | 每单处理毫秒数（API 与可选 CLI 演示） | `10000` |
| `FEEDME_DEMO_FAST` | `run-demo` 是否缩短处理时间（仅非交互演示） | 未设置则与上者一致；CI 可设为 `1` 配合更短毫秒 |

## 5. CLI 实施约定

### 5.1 `run-demo`（供 `run.sh` / CI）

- **必须**将日志写到 **标准输出**，由 `run.sh` 重定向到 `scripts/result.txt`。
- 每行或关键行包含 **`HH:MM:SS`**（24 小时制）；推荐格式 **`[15:04:05]`** 或 **`15:04:05`**，与现有示例 [scripts/result.txt](../scripts/result.txt) 风格接近即可通过 grep。
- 内容需 **有意义**（作业 Submission）：至少包含创建订单、Bot 动作、完成/取消等事件，而非单行占位。
- 退出码：`0` 表示演示跑完；非 `0` 便于 CI 失败（若 workflow 将来增加 `set -e` 链）。

### 5.2 `interactive`（面试）

建议最小命令集（实现时可调整，但需自洽）：

| 输入 | 行为 |
|------|------|
| `n` | New Normal Order |
| `v` | New VIP Order |
| `+` | Add Bot |
| `-` | Remove newest Bot |
| `s` | 在终端用 **英文** 打印当前状态（文本表） |
| `q` | 退出（提示符与帮助文案建议英文） |

要求：**阻塞读 stdin**、有明确 **英文** 提示符与 `help`；与 `run-demo` **不同入口**（子命令），避免 CI 卡住。

## 6. Angular UI 实施建议（对齐 README 用词，全英文界面）

- **用户可见字符串一律英文**（按钮、分区标题、状态、空状态提示）；组件 `.ts` / `.html` 旁的实现说明注释可用 **中文**。
- **按钮文案**与 README 一致：**New Normal Order**、**New VIP Order**、**+ Bot**、**- Bot**。
- **区域标题**：**PENDING**、**COMPLETE**；处理中可在 **PROCESSING** 区展示，或在 **Bot** 卡片上显示当前 **Order #**（需能观察 R4/R5）。
- **Bot 列表**：**ID**、状态 **IDLE** / **Processing**、当前订单号（若有）；勿使用中文状态词作为 UI 文案。
- **轮询**：间隔约 **1s**；组件销毁时 `clearInterval` / `unsubscribe`，避免泄漏。
- **生产环境 API 基地址**：使用 **相对路径 `/api`**（与 Nginx 同源），避免硬编码域名；`environment.development.ts` 可指向 `http://localhost:8080`。

## 7. 领域层实现检查清单（防竞态）

- `-Bot`：**先取消**正在执行的处理上下文，**等待**处理 goroutine 退出（或等价同步），再 **从 Bot 列表移除** 并 **将订单插回 PENDING**；避免「已移除 Bot 仍完成订单」写入 COMPLETE（与 [架构与接口设计.md](架构与接口设计.md) §2.4 一致）。
- 定时器与 `Done` **同时就绪**时：在「标记 COMPLETE」前 **再次检查** `ctx.Err()`。
- **无 Bot 时 `-`**：不崩溃；HTTP 返回约定错误码；CLI interactive 打印提示。

## 8. 验收剧本（建议手测顺序）

下列顺序可用于本地与面试演示，覆盖 R1–R7 与 VIP 穿插：

1. 新建 Normal #1 → 仅 PENDING Normal。
2. 新建 VIP #2 → PENDING 顺序为 VIP#2、Normal#1。
3. `+Bot` → Bot1 处理 #2；约 10s 后 #2 COMPLETE，Bot1 再处理 #1。
4. 再建 VIP #3、Normal #4；`+Bot` ×2 → 观察两 Bot 并行、VIP 优先于 Normal。
5. 某 Bot 处理中执行 `-Bot`（最新）→ 该单回到 PENDING，**相对顺序**符合架构约定。
6. 清空 PENDING 后 Bot 显示 **IDLE**；再下一单 → 立即被取走。
7. 重启服务 → 状态清空（R7）。

（订单号为示意；实际以递增 ID 为准。）
