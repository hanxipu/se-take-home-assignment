# 本地测试说明

本文档描述 **在实现代码之后** 如何在本地验证功能与脚本。**当前 GitLab 不可用**，不以流水线为准；以本机命令与人工检查为主。

## 1. 环境准备

| 组件 | 建议版本 | 说明 |
|------|-----------|------|
| Go | 与 [.github/workflows/backend-verify-result.yaml](../.github/workflows/backend-verify-result.yaml) 接近（如 1.23.x） | 避免语言特性不兼容 |
| Node.js / npm | 与 workflow 中 Node 22.x 同主版本更佳 | Angular CLI 与构建 |
| Git Bash 或 WSL | Windows 用户 | `scripts/*.sh` 为 bash；PowerShell 不直接执行 `./scripts/test.sh` |
| Docker Desktop | 可选 | 本地模拟 Nginx + API 容器 |

## 2. Go 后端与脚本

在仓库根目录执行（路径按实现后调整）：

```bash
# 单元测试
go test ./... -v

# 或使用作业脚本（需 chmod +x，Linux/macOS/Git Bash）
chmod +x scripts/test.sh scripts/build.sh scripts/run.sh
./scripts/test.sh
./scripts/build.sh
./scripts/run.sh
```

**验收 `scripts/result.txt`**

- 文件存在且非空。
- 至少一行包含 `HH:MM:SS` 形式时间戳（与 CI 中 `grep -E '[0-9]{2}:[0-9]{2}:[0-9]{2}'` 一致）。

可在本地模拟 CI 检查：

```bash
test -s scripts/result.txt && grep -E '[0-9]{2}:[0-9]{2}:[0-9]{2}' scripts/result.txt
```

## 3. HTTP API 单独验证

实现 `cmd/server` 后（示例）：

```bash
# 启动服务（端口以实际为准，如 8080）
./server   # 或 go run ./cmd/server

# 另开终端
curl -s -X POST http://localhost:8080/api/orders -H "Content-Type: application/json" -d '{"kind":"vip"}'
curl -s http://localhost:8080/api/state
```

按 [架构与接口设计.md](架构与接口设计.md) 逐项验证 **R1–R7**：含 -Bot 回队顺序、每单 **10 秒** 完成（可用环境变量缩短仅用于开发自测）。

**R7（无持久化）**：重启 API 进程（或容器）后，订单与 Bot 状态应 **清空或恢复初始**，不依赖数据库/磁盘即为通过。

## 4. Angular 前端

```bash
cd frontend
npm ci
npm start
# 或 ng serve，默认 http://localhost:4200
```

- 开发环境将 API 基地址指向 `http://localhost:8080`（或经代理），具体以 `environment.ts` / `proxy.conf` 为准（实现时补充）。
- 浏览器中手动点击：普通单、VIP、+Bot、-Bot，观察三区列表与 Bot 状态是否与后端快照一致。
- 界面文案应为 **英文**（与根目录 README 作业描述一致）；详见 [实施方案.md](实施方案.md) §4。

## 5. 容器本地联调（可选）

在提供 `docker-compose.yml` 后：

```bash
docker network create proxy   # 若尚未创建，与线上一致
docker compose up --build
```

浏览器访问映射端口或本机 hosts，确认静态页与 `/api` 反代正常。

## 6. Windows 常见问题

- **`./scripts/run.sh` 无法执行**：使用 Git Bash、WSL，或将命令手动翻译为 PowerShell（不推荐长期维护两套，以 bash 为准）。
- **端口占用**：更换 `server` 监听端口或结束占用进程。
- **防火墙**：首次允许 Node/Go 访问网络。

## 7. 与 CI 的关系

GitHub Actions 会在 PR 上顺序执行 `test.sh` → `build.sh` → `run.sh` 并校验 `result.txt`。**默认工作流不包含 `ng build`**，Angular 能否生产构建需在本地执行 `ng build`（或见 [持续集成说明.md](持续集成说明.md) 中的可选增强）。**当前阶段不跑 CI 时**，以上本地步骤即为预检清单；恢复 CI 后仍应优先保证本地通过。
